步骤 1：找黑客成功登录的 IP 地址（Flag：登录 IP）

先看安全日志

为什么看Security.evtx（安全日志）？

因为 Windows 会把 “用户登录 / 注销” 这类涉及安全的操作，专门记录在安全日志里（这是系统默认的审计规则）。

为什么用「事件 ID 4624」？

“事件 ID” 是 Windows 给每种操作的 “编号”——事件 ID 4624 = 成功的账户登录(对应的，事件 ID 4625 就是 “登录失败”）。要找 “成功登录”，就筛选这个 ID。

操作流程 & 日志

（1）操作步骤

  1. 打开「Security.evtx」；

  2. 筛选 “事件 ID 4624”（只看成功登录的记录）；

  3. 按 “时间排序”（从新到旧或旧到新,找到可疑IP）；

  4. 看日志里的 “网络信息”，找到「源网络地址」（就是黑客的 IP）。

（2）关键日志片段解释

「工作站名」：是被攻击的服务器名称，不是黑客的；

「源网络地址」：字面意思 “来源的网络地址”，就是黑客电脑的 IP，这就是我们要的关键信息；

「源端口」：黑客电脑上随机分配的端口，解题用不上，但知道是 “登录连接的端口” 即可。

4. 最终 Flag

黑客登录 IP 是192.168.36.133，所以 Flag：flag{192.168.36.133}

步骤 2：找黑客修改后的用户名（Flag：新用户名）

账户修改（改用户名、改密码、禁用账户等）属于 “安全审计事件”，依然记录在安全日志(Security.evtx)里。

事件 ID 4738 = 用户账户被修改（比如改用户名、改显示名、改邮箱等），这个 ID 会明确记录 “被修改的账户是谁”“修改前是什么值”“修改后是什么值”。

（1）操作步骤

  1. 还是在Security.evtx里；
  2. 筛选 “事件 ID 4738”（只看账户修改记录）；
  3. 看日志里的「TargetUserName」（被修改的账户名）和「SamAccountName」（Windows 账户的 “短名”，核心标识）。

（2）

日志里有两个核心字段需要区分：

SubjectUserName：执行修改操作的人（比如 “谁改了用户名”），这里显示为WIN-B1B1T3K57G9$（带$的是系统 / 计算机账户，说明是黑客通过系统权限改的）；

TargetUserName SamAccountName：被修改的账户名（也就是我们要的 “修改后的用户名”），这两个字段都显示为Adnimistartro（注意不是默认的Administrator，多了个r，是黑客改的）。

为什么这两个字段能确定？SamAccountName是 Windows 账户的 “唯一短名”（比如默认管理员是Administrator），一旦修改，这个字段会直接显示新值；TargetUserName是 “目标账户名”，和SamAccountName一致，就说明这是修改后的最终用户名。

修改的日志不多,一个个尝试即可,最后发现最可疑的就是Adnimistartro,与默认管理员Administrator相似

4. 最终 Flag

修改后的用户名为Adnimistartro，所以 Flag：flag{Adnimistartro}

步骤 3：找黑客访问的关键文件名（Flag：文件名，无后缀）

还是看Security.evtx

“访问系统关键文件” 属于 “对象访问审计事件”，安全日志会记录这类高权限操作（前提是系统开启了文件审计，但靶场里已经开启，所以有记录）。

事件 ID 4663 = 尝试访问对象（文件 / 文件夹 / 注册表等），这个 ID 会明确记录 “谁访问了哪个文件”“用什么进程访问的”“有没有权限”。

（1）操作步骤

  1. Security.evtx中筛选 “事件 ID 4663”；

  2. 看日志里的ObjectType(对象类型)是否为File（确保是文件，不是文件夹或注册表）；

  3. 看ObjectName（文件路径），找 “系统关键路径” 下的文件（比如C:\Windows\System32\下的文件，普通用户不会随便访问）；

  4. 日志看着很多其实大多是重复的,一个个尝试特别是system路径下的,最后发现是SCHEMA.DAT(用户名是administrator是因为黑客先查看文件后改动的用户名,可以查看对应时间就知道)

  5. 提取文件名（不含后缀）。

    事后查询一下为什么是它

SubjectUserName: administrator：说明是高权限账户访问（黑客控制了管理员账户）；

ObjectName：文件路径是C:\Windows\System32\...（System32 是系统核心目录，里面的文件很关键），文件名是SCHEMA.DAT；

ProcessName: explorer.exe：是 Windows 的 “文件管理器”，说明黑客手动打开了这个文件（不是系统自动访问，更可疑）。

SCHEMA.DAT是 Windows 的 “系统架构配置文件”，记录系统组件的配置信息，普通用户几乎不会访问，只有高权限操作（比如黑客篡改系统配置）才会动它，所以是 “关键文件”。

4. 最终 Flag

文件名是SCHEMA.DAT，不含后缀就是SCHEMA，所以 Flag：flag{SCHEMA}

步骤 4：找最后一次重启数据库的进程 ID（Flag：进程 ID）

MySQL 是 “应用程序 / 服务”，它的启动、停止、报错等事件，会记录在应用日志里（系统日志只记录操作系统自身的服务，比如 Windows Update，不记录 MySQL）。

不同应用的事件 ID 是自己定义的，对于 MySQL 来说，

事件 ID 100 = MySQL 服务启动

（重启本质是 “停止 + 启动”，所以找启动事件就能拿到重启后的进程 ID）。

（1）操作步骤

  1. 打开Application.evtx；

  2. 筛选 “事件 ID 100” 且 “事件来源（Provider）为 MySQL”；

  3. 按时间排序，找 “最后一次” 的启动事件；(通过之前的题目可知黑客在日期10.8攻击,所以时间跨度太大的就不看比如12月份的,通过查看EventData中的Data排除其他的日志,找第一个符合要求的即可)

  4. 从日志中提取 “进程 ID”（starting as process 8820）。

（2）关键 XML 日志片段解释（靶场用 XML 视图查看）

  • Provider Name="MySQL"：确保这个事件是 MySQL 产生的，不是其他应用；
  • TimeCreated：按时间排序后，这个时间是 “最后一次” 启动，所以对应的进程 ID 是最后一次的；
  • EventData中的Data：直接写了 “starting as process 8820”，翻译过来就是 “以进程 ID 8820 启动”—— 这就是我们要的进程 ID。

4. 最终 Flag

最后一次重启 MySQL 的进程 ID 是8820，所以 Flag：flag{8820}

步骤 5：找修改后用户重启系统的次数（Flag：次数）

黑客修改用户名（步骤 2 的Adnimistartro）后，多次重启系统，需要统计用这个修改后用户名重启的次数。

系统的 “启动 / 重启 / 关机” 是操作系统核心操作，记录在系统日志里（应用日志不记录这个）。

系统重启分两种：

系统自动重启（比如更新后）：用事件 ID 6005/6006（日志服务启动 / 停止，间接表示系统启动 / 关机）；

用户 / 进程手动触发的重启 / 关机：用事件 ID 1074，这个 ID 会明确记录 “谁触发的重启”（用户名），所以必须用 1074 才能关联到 “修改后的用户名”。

3. 操作流程

（1）操作步骤

  1. 打开System.evtx；
  2. 筛选 “事件 ID 1074”；
  3. 排除 “关机”“关闭电源”（只留 “重启” 操作）；
  4. 排除 “用 Administrator（原用户名）”“用 SYSTEM（系统账户）” 触发的重启；
  5. 统计 “用 Adnimistartro（修改后用户名）” 触发的重启次数。

（2）关键筛选逻辑

第一步：区分 “重启” 和 “关机”事件 ID 1074 的描述里会写 “操作类型”，比如：

“关机”“关闭电源”：不算，题目要的是 “重启”；

“重新启动”：算，保留这类记录。

第二步：区分 “修改前用户” 和 “修改后用户”日志里会有「User Name」字段（触发重启的用户名）：

若为Administrator（原管理员名）：不算（还没修改用户名）；

若为SYSTEM（系统账户）：不算（不是黑客手动操作）；

若为Adnimistartro（修改后用户名）：算，统计次数。

（3）靶场实际统计结果

靶场里筛选后，符合 “修改后用户（Adnimistartro）+ 重启” 的记录共3 次（排除了关机、关闭电源、用原用户名 / SYSTEM 的重启）。

也可以不用筛选,日志不多,一个个查看即可

这里有一个小细节,我们单查看日志用户名发现只有两次,但是看时间

黑客在14.07就已经掌控了主机,所以这个system的重新启动也是黑客的通过系统程序重新启动的,加上后面两次一共三次

4. 最终 Flag

重启次数是3，所以 Flag：flag{3}
